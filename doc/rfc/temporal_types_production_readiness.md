# Production-readiness scoreboard for MobilityDB temporal types

A single-page audit of every temporal type in MobilityDB against the
gates I treat as "production-ready":

1. **DocBook chapter** in `doc/temporal_*.xml` integrated into
   `doc/mobilitydb-manual.xml`.
2. **SQL regression tests** under `mobilitydb/test/<type>/queries/`,
   with matching expected outputs and zero entries in `SKIP_TESTS`.
3. **No commented-out `extern`** in `meos/include/meos_*.h` —
   anything declared in the header is implemented in
   `meos/src/<type>/`.
4. **MEOS smoke suite** (auto-generated by
   `meos/test/gen_smoketest.py`) for valgrind-style leak detection
   without the PostgreSQL infrastructure masking it.
5. **Production-guidance sect1** in the type's DocBook chapter
   (covering when-to-use, hazards classified per the four-tier
   A/B/C/D rule, durability and storage), with an inbound cross-link
   from the chapter introduction.

## Per-type gap detail

### tpose
- `tpose_orientation` is declared but commented out in
  `meos/include/meos_pose.h:230`. Every other `tpose_*` function is
  implemented and tested. The orientation surface likely needs a
  3D-aware variant (rotation alone is enough for 2D); needs a design
  pass before an implementation lands.

### tcbuffer
- `160_tcbuffer_distance_tbl` — ✅ reactivated on fork branch
  `ci/reactivate-160-tcbuffer-distance-tbl`. Master commit `ee27da1a6`
  aligned the NAD failure sentinel; the test now passes immediately.
- `162_tcbuffer_spatialrels[+_tbl]` — ✅ reactivated on fork branch
  `ci/reactivate-162-tcbuffer-spatialrels`. Expected outputs were
  missing; generated against current master, both forms pass.
- `164_tcbuffer_tempspatialrels[+_tbl]` — ⏳ blocked on PR
  `MobilityDB#847`. The test calls `tDwithin(geometry, tcbuffer, ...)`
  which doesn't exist; #847 enables those overloads and ships the
  expected outputs.
- After #847 merges and the three `ci/reactivate-*` branches land,
  tcbuffer goes ✅.

### trgeometry
- `122_trgeo`, `124_trgeo_compops[+_tbl]` — ⏳ blocked on PR
  `MobilityDB#849` (cross-type lifting infra crash on
  trgeo↔geometry compops).
- 8 commented-out externs in `meos/include/meos_rgeo.h`:
  - 6 of them (`trgeo_at_geom`, `trgeo_minus_geom`, `trgeo_at_stbox`,
    `trgeo_minus_stbox`, `trgeo_traversed_area`, …) are implemented
    on fork branch `feat/trgeo-spatial-restrictions` (with adaptive
    sub-sampling for `traversed_area`).
  - 2 remaining (`trgeo_at_geo`/`trgeo_minus_geo` for geography
    inputs, `trgeo_at_elevation`/`trgeo_minus_elevation` for height
    restrictions) are scoped feature work; not blocking the
    mainstream production surface but need design.
- DocBook chapter shipped on fork branch `doc/trgeo-docbook-chapter`
  (full reference, ~550 lines, mirrors the depth of
  `temporal_poses.xml`).
- Once #849 merges and the four trgeo fork branches
  (`feat/trgeo-distance`, `feat/trgeo-distance-tail`,
  `feat/trgeo-spatial-restrictions`, `doc/trgeo-docbook-chapter`)
  land, trgeometry goes ✅ for the mainstream surface.

## Production-guidance documentation surface

A separate work axis (gate 5) covers the DocBook production-guidance
sect1 — operational guidance, hazard taxonomy per the four-tier
A=ERROR / B=NOTICE-WARNING / C=docs-only / D=docs-only rule, and
durability / storage conventions — with an inbound cross-link from
the chapter introduction so a reader landing in the type taxonomy can
navigate to the operational material end-to-end.

All seven tracked types now have both halves on the fork. The branches
have been **squashed for review** — each fork branch is a single
well-messaged commit on top of `upstream/master`, except where the
work warranted splitting into multiple PRs by concern.

The geo branch is the template (sect1 + inbound cross-link + a
hazard regression suite in `meos/test/geo_test.c`); the other
single-PR branches ship the sect1 + cross-link, with the hazard
regression suite still to land per branch.

The original `pose/production-readiness` and
`tnpoint/production-readiness` branches are kept on the fork as
historical-reference backup; they are superseded by the split
branches and should not be opened for review.

The `docs/tnpoint-production-guidance` branch is **stacked on**
`feat/tnpoint-mfjson` rather than rooted on `upstream/master` — the
production-guidance sect1 references the MFJSON shape and field
names introduced by the MFJSON PR, so the docs PR depends on the
MFJSON PR landing first. The dependency is captured in the PR body.

The `docs/pose-production-guidance` branch additionally ships a
`pose_mfjson_conformance` sect2 documenting the OGC GeoPose
Basic-YPR conformance shape and the SRID-labelling design choice;
the `docs/tnpoint-production-guidance` branch documents the
int64-precision JS-consumer caveat in `tnpoint_durability`. Both
are tier-D design choices in the four-tier hazard rule.

## Path to all-types production-ready

In dependency order:

1. `MobilityDB#847` merge → unlocks tcbuffer 164 reactivation.
2. `MobilityDB#849` merge → unlocks trgeo 122/124 reactivation.
3. Land the fork branches in any order:
   - `ci/reactivate-160-tcbuffer-distance-tbl`
   - `ci/reactivate-162-tcbuffer-spatialrels`
   - `feat/trgeo-distance` + `feat/trgeo-distance-tail`
   - `feat/trgeo-spatial-restrictions`
   - `doc/trgeo-docbook-chapter`
   - `test/trgeo-meos-suite`, `test/meos-smoke-all-types`,
     `test/cbuffer-npoint-smoke-fix`, `test/meos-geo-smoke`,
     `test/meos-smoke-ci`
   - `fix/meos-bug-tpose-to-tpoint`,
     `fix/trgeo-spatialrels-h-missing-semicolons`
4. Optional: design pass for `tpose_orientation`,
   `trgeo_at_geo`/`trgeo_minus_geo`, `trgeo_at_elevation`/
   `trgeo_minus_elevation` to close the last commented-out gaps.

After (1)–(3), every type in this scoreboard is ✅. The optional
items in (4) are minor extensions, not blockers — they can ride
later releases.

The production-guidance sect1 (gate 5) lands on a separate per-type
PR cadence — independent of (1)–(3) — once the per-type fork
branches listed above are signalled review-ready. The hazard
regression test for each type (currently shipped only for geo) is
the natural next commit on each per-type branch before it is
opened for review.
